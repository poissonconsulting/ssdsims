## Why

`TARGETS-DESIGN.md` ¬ß4 describes the package's whole point: take a scenario object that already runs locally and dispatch its shards across a SLURM cluster, changing **only** the `crew` controller and its resource specs ‚Äî pipeline shape (from ingredient A) and task content + RNG (from ingredient C) are scheduler-independent. The crew labs (sister planning repo) have already assembled and run that shape end to end: a bare `crew.cluster::crew_controller_slurm()` dispatching a no-op workload (A), the operational backend it needs ‚Äî a ManyLinux binary install path plus a login-node prerequisite checker (B) ‚Äî and the real ssdsims per-sim pipeline driven through that same controller (C). So `cluster-pipeline` (¬ß12) **ports a proven shape, not an untested sketch**.

Today the package ships `inst/targets-templates/small/` and `large/`; the `large/` template only *gestures* at this ("swap in `crew.cluster` controllers for SLURM/PBS ‚Ä¶ (`cluster-pipeline`)") with a local `crew::crew_controller_local()` controller. There is no `cluster/` template, no `crew_controller_slurm()` wiring, and `crew.cluster` is not in `DESCRIPTION`. A user who wants to run a study on a cluster has to invent the controller block from scratch ‚Äî exactly the wiring the labs already de-risked.

## What Changes

- Add `inst/targets-templates/cluster/` (the editable template, resolving ¬ß11 Q5 in favour of *ship a template*, not docs-only): a `_targets.R` that builds the scenario's pipeline via the **existing** `ssd_scenario_targets()` factory (the shape, A; the content, C ‚Äî both unchanged from `large/`), but sets `tar_option_set(controller = crew.cluster::crew_controller_slurm(...))` (the backend, B) with editable `script_lines` (module loads), `slurm_partition`/queue, and scratch/worker config. The factory call and scenario are scheduler-independent and reused verbatim; only the controller block is cluster-specific.
- A controller block that the user edits per cluster: queue/partition, `script_lines` (e.g. `module load R/4.x`), scratch/`tempdir`, worker count, and walltime ‚Äî drafted from the labs' working block (B), not invented.
- A **probe run before any scenario logic** that validates *both* connectivity and worker prerequisites: it submits and reclaims one Slurm job through the controller and, inside that job, verifies R resolves, `library(ssdsims)` loads (the ManyLinux binary path), and scratch/`tempdir` is writable, returning a small witness. It is an **explicit upstream dependency of the first scenario shard**, so `tar_make()` builds it first and a failure aborts with an actionable message (which check failed ‚Üí which fix: queue/account wiring, module/install path, or storage) rather than letting the shards run.
- **Documentation that takes a user from zero to a running cluster job**: a guide that maps the cluster's own (non-R) SLURM usage instructions ‚Äî login, `sbatch`, partition/queue, account, `module load`, scratch, walltime ‚Äî onto the `crew_controller_slurm()` arguments, then runs the built-in `small` scenario end-to-end as the minimal first job, then adapts to the user's own scenario. Most users start from site IT instructions that say nothing about R/`crew`/`targets`; the docs bridge that gap.
- A `run.R` driver mirroring `large/run.R`: source the scenario, `tar_make()`, report the summary path and a peek at the estimates ‚Äî plus a guard that **skips/aborts cleanly** when `crew.cluster` is not installed or no Slurm queue is reachable, falling back to a `crew::crew_controller_local()` smoke path off-cluster (mirroring the labs' fallback).
- Add `crew.cluster` to `DESCRIPTION` `Suggests` (this proposal *describes* the addition; it is applied by the implementing change, listed in tasks).
- End-to-end validation: `tar_make()` on a real (or sandboxed) Slurm queue builds the probe then the scenario shards, byte-identical to the local pipeline's per-task results.

## Capabilities

### New Capabilities
- `cluster-pipeline`: an editable cluster targets template that builds a scenario via the shared `ssd_scenario_targets()` factory under a SLURM `crew.cluster` controller, validates cluster connectivity *and worker prerequisites* before scenario logic, ships a "zero to a running cluster job" guide bridging the site's own (non-R) SLURM instructions to the controller and a minimal `small` run, dispatches shards as the unit of parallelism across Slurm jobs, and degrades cleanly when `crew.cluster` or a Slurm queue is unavailable.

## Impact

- **New files**: `inst/targets-templates/cluster/` ‚Äî `_targets.R`, `scenario.R`, `run.R` (and a `run-serial.R` mirror for the off-cluster comparison, matching the `small/`/`large/` shape); plus a "zero to a running cluster job" guide (a vignette/pkgdown article, or the template README) carrying the site-instruction ‚Üí `crew_controller_slurm()` mapping table and the minimal `small`-scenario walkthrough.
- **Dependencies**: adds `crew.cluster` to `DESCRIPTION` `Suggests` (`crew` is already there). Both stay in `Suggests` ‚Äî the cluster path is opt-in and the package builds/tests without a scheduler.
- **No R code or live-spec changes**: the pipeline is assembled by the existing `ssd_scenario_targets()` factory (`task-shards`); this change adds only template files and the controller block around it. Scheduler-independence (¬ß4) is the reason no factory change is needed.
- **Dependencies (direction)**: depends on `task-shards` (the `ssd_scenario_targets()` factory) and the scenario (`ssd-define-scenario`); it is **not** a prerequisite of any other step. `TARGETS-DESIGN.md` ¬ß12 DAG node: `task-tables ‚Üí cluster-pipeline` (the factory must exist; the template wraps it).
- **Resolves open questions**: ¬ß11 Q5 (ship the editable `inst/targets-templates/cluster/` template, not docs-only) and Q6 (commits only to *shards being the unit of parallelism*; the precise shard-target ‚Üî Slurm-job packing is a documented `crew` configuration knob, recorded in this template's prototype).
- **When to land it**: any time after `task-tables`/`task-shards` (the factory) lands; operationally, when the first study is to be run on a cluster. Not needed for the local/small/large templates.
