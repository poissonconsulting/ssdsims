## Context

`ssd-define-scenario` landed the declarative `ssdsims_scenario` object (`TARGETS-DESIGN.md` Â§1): seed, scenario options, dataset names, fit/hc argument grids, `partition_by`, and the recorded `ci = FALSE` ignore. Nothing consumes it yet. The roadmap (Â§12) makes `task-list-loop-baseline` the immediate next step on the `define â†’ baseline` edge: turn the scenario into the **flat per-task data shape** that all later steps build on, plus a no-frills runner.

The Â§2 "What goes into the hash" discussion already specifies the per-step identity axes â€” data: `(dataset, sim, replace)`; fit: data-identity + fit-grid row; hc: fit-identity + hc-grid row (modulo the Â§1.2 collapse). This step materializes exactly those tables as plain tibbles, with **no** RNG/primer columns and **no** targets/shard machinery. The point is to freeze the data shape early so subsequent steps (`task-primer`, `state-primitives`, `task-tables`) each change one thing against a working baseline rather than introducing the shape and the shards and the RNG all at once.

## Goals / Non-Goals

**Goals:**
- Three derivation functions producing `data`, `fit`, and `hc` task tables (tibbles) from a scenario, one column per cross-join axis.
- Honour the Â§1.2 `ci = FALSE` collapse at hc-table expansion using the flag the scenario already records.
- A baseline runner: three `purrr::pmap()` loops, data â†’ fit â†’ hc, threading outputs forward, returning collected results.
- Establish the column contract that `task-tables` later decorates with `(seed, primer)` and groups into shards.

**Non-Goals:**
- No RNG, no `(seed, primer)`/`stream` columns (those land in `task-primer` / `state-primitives`).
- No `targets` dependency, no `tar_map`, no shards, no `partition_by` grouping (those land in `task-tables`).
- No Hive partitioning, no Parquet I/O (later steps).
- No sub-truncation behaviour for `nrow` (the Â§5 property; here `nrow` is merely carried as a column).
- No change to `ssd_run_scenario()` or the existing public pipeline.

## Decisions

- **Decision: four separate tables, not one wide join.** The pipeline fans out in four steps with different fan-outs and (later) different shard axes (Â§5 `partition_by` defaults differ per step). Keeping `sample`/`data`/`fit`/`hc` as distinct tibbles, each referencing its parent by an id foreign key, mirrors the Â§6 step structure and lets `task-tables` group each independently. *Alternative*: a single fully-joined table â€” rejected; it conflates the per-step shard boundaries and bloats rows.
- **Decision: sampling is its own task type; `nrow` is an axis of truncation, not of the draw.** Reviewer-driven (PR #80). Â§5 proves every `nrow` value is a sub-truncation of one `n_max`-row draw. Rather than carry `nrow` as a compound column on a conflated `data` step, the draw becomes a distinct `sample` task keyed `(dataset, sim, replace)` carrying `n_max`, and a separate RNG-free `data` step truncates it to each `nrow` (`head(sample, nrow)`). So `nrow` is a clean scalar cross-join axis from the `data` step down, the expensive draw is shared *structurally* (one `sample` task per `(dataset, sim, replace)`), and the Â§5 efficiency holds without a carried-column trick. *Alternative A*: carry `nrow` as a list column on a single `data` step â€” rejected (the compound column the review flagged). *Alternative B*: carry a scalar `n_max` on a single `data` step and truncate at consumption â€” rejected; it leaves `nrow` with no clean home as an axis and hides the truncation. *Alternative C*: make `nrow` a cross-join axis of the *draw* â€” rejected; it re-draws redundantly and contradicts Â§5.
- **Decision: explicit path-style `<step>_id` keys and parent foreign keys.** Reviewer-driven (PR #80). Â§5 anticipates `data_id` / `fit_id` columns as "sugar for the path". Each table carries a path-style primary key built from its cross-join axes (the Â§6 Hive partition path, e.g. `dataset=boron/sim=1/replace=FALSE`) plus its parent's id as a foreign key, so a child references its parent by one joinable column and the runner threads via the FK. The id is deterministic and stable under scenario growth (adding a dataset does not renumber existing rows), matching the Â§8 cache-by-existence story. *Alternative*: sequential integer ids â€” rejected; they renumber on growth. *Alternative*: implicit composite key (carried identity columns only) â€” rejected; the reviewer wants an explicit single-column foreign key, and Â§5 calls for the path id.
- **Decision: `replace` is a first-class scenario option.** Reviewer-driven (PR #80). `replace` is a sampling axis on a par with `nrow`, already an argument on `ssd_sim_data()` / `ssd_run_scenario()`. `ssd_define_scenario()` gains a validated `replace` argument (logical, length â‰¤ 2, default `FALSE`) and the `sample` derivation reads `scenario$replace`. *Alternative*: default it to `FALSE` inside the derivation â€” rejected; it hid a real axis and the scenario is the single source of truth for the grids.
- **Decision: the scenario retains the data it was built from.** Reviewer-driven (PR #80). The scenario keeps the validated `ssd_data()` collection in `$data` (and `$datasets` keeps the names), so a local run reads the data straight off the scenario and `ssd_run_scenario_baseline(scenario)` needs no `data` argument. The names remain what the targets/cluster path hashes and resolves through the registry (Â§1.1), so that path still need not carry the data frames. *Alternative*: keep the scenario a names-only manifest and pass `data` to the runner â€” rejected; redundant for local use and the scenario is meant to be self-sufficient. *Alternative*: a registry now â€” out of scope (future step).
- **Decision: reuse the scenario's recorded `ci = FALSE` collapse.** The collapse is decided at construction time and recorded on the scenario (`ssd-define-scenario` Â§1.2). The hc derivation honours that flag rather than re-deriving the rule, keeping a single source of truth. *Alternative*: re-implement the collapse here â€” rejected; duplicates logic and risks drift.
- **Decision: `min_pmix` carried by name.** Per Â§1.1/Â§2, function-valued scenario options are referenced by name so later hashing is stable. The fit table stores the name; value lookup is a later step's concern. *Alternative*: embed the function â€” rejected; defeats the name-only indirection.
- **Decision: `tidyr::expand_grid()` / `dplyr::cross_join()` for the cross-joins; `purrr::pmap()` for the runner.** Both are already in `Imports` and match the Â§12 bullet ("`purrr::pmap()` loops"). *Alternative*: a `targets` plan â€” rejected; explicitly out of scope and a later step.
- **Decision: task tables are a classed tibble (`ssdsims_tasks`) with a `print()` method; a compound `ssd_scenario_tasks()` bundles all four into an `ssdsims_task_set`.** A raw tibble prints as an undifferentiated grid; a thin S3 subclass that records the step lets `print.ssdsims_tasks()` lead with the step name, the cross-join axes, and the task count before a compact row preview â€” matching `print.ssdsims_scenario()`. Subclassing a tibble keeps every dplyr/tidyr verb working (they operate on the underlying tibble; the class is reattached or harmlessly dropped). The compound `ssd_scenario_tasks()` is the canonical expansion entry point (Â§1/Â§2) and returns an `ssdsims_task_set` exposing the four independently-classed tables (so `task-tables` can still group each on its own shard boundary). *Alternative*: return bare tibbles â€” rejected; the step/axis metadata and the nice printing have nowhere else to live.
- **Decision: the runner is "stupid" â€” it consumes the derived tables, it does not expand.** Reviewer-driven (PR #80). The runner calls `ssd_scenario_tasks()` and loops over each table with `purrr::pmap()`, resolving each task's parent result by the parent's id foreign key (no in-runner cross-join, no key reconstruction). It reuses the per-step operations (slice-sample / `head()` / `ssd_fit_dists` / `ssd_hc`) without RNG seeding, in plain order. The baseline does not need reproducibility yet; `state-primitives` introduces per-task seeding on top of this shape. *Alternative*: re-cross-join the grids inside the runner â€” rejected; it duplicates the derivation and is exactly what the review flagged.

## Risks / Trade-offs

- **Shape churn in later steps** â†’ Mitigated by deriving the axes directly from Â§2's hash contract, so `task-tables` adds columns (`seed`, `primer`) rather than reshaping. Snapshot tests pin the column contract so later changes are visible diffs.
- **`ci = FALSE` collapse divergence from the scenario** â†’ Mitigated by consuming the scenario's recorded flag rather than re-deriving; a test asserts the hc table row count matches the Â§1.2 example grid.
- **Baseline runner is not reproducible (no seeding)** â†’ Accepted and intentional for this step; documented as a non-goal. Reproducibility arrives with `state-primitives`. Tests for the runner assert structure/threading, not draw values.
- **`sample` vs `data` naming could confuse readers** â†’ Mitigated with roxygen and an inline comment: `sample` is the `n_max`-row draw keyed `(dataset, sim, replace)`; `data` is its `head(., nrow)` truncation. Pulling the truncation split forward also partly implements the `nrow-sub-truncation` roadmap item â€” accepted, because freezing a 3-step shape that step would tear out is the churn this baseline exists to avoid.
- **Four-type shape revises the north star** â†’ `TARGETS-DESIGN.md` Â§5 is updated to describe `sample`/`data`/`fit`/`hc` and the path-id foreign keys; the partition_by defaults and target graph are reconciled in the later `task-tables` step.
