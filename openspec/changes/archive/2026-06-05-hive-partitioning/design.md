## Context

`TARGETS-DESIGN.md` ┬º6 splits `hive-partitioning` into two halves: a **storage/read** half (one Parquet per `partition_by` path cell, `format = "file"` outputs, the m:n parent-shard read) and a **caching/invalidation** half (┬º8 ÔÇö how a scenario change or a code edit decides which shards rebuild). The storage/read half has **already landed** in the live `task-shards` and `shard-runner` specs and in `R/targets-runner.R`: `ssd_run_{sample,fit,hc}_step()` write byte-stable Parquet shards, `read_parent_shards()` opens the distinct set of parent shards a child's tasks reference (resolving the m:n fan-in with duckplyr predicate pushdown), and `ssd_scenario_targets()` mints one named `format = "file"`, `error = "null"` target per shard via `tarchetypes::tar_map()`, with results byte-identical to `ssd_run_scenario_baseline()`. This change is **re-scoped to the caching/invalidation half only** and does not touch the storage/read code or its specs.

The gap it closes: the shipped factory wires step ordering with **coarse** `tarchetypes::tar_combine()` barriers (`sample_done` / `fit_done` / `hc_done`) and sets **no `tar_cue()`**. Each child step's shard targets reference the whole-step barrier, so under `targets`' default content-hash propagation **a single rewritten parent shard marks the entire downstream step out of date** ÔÇö every child shard re-runs, not just the ones that read the rewritten parent. And ┬º8 records two undecided forks every downstream incremental-rebuild step waits on: the **invalidation model** (cache-by-existence vs. content-hash; the `tar_cue` pinning semantics) and the **`data`-step-vs-fold** decision, whose answer ┬º8 ties directly to which model is chosen.

## Goals / Non-Goals

**Goals:**

- Pin the shard step targets' invalidation model as an explicit `task-shards` requirement (not an emergent `targets` default): content-hash propagation over the `format = "file"` shard outputs, the cache-by-existence reading of a shard's identity (its partition path), and the `tar_cue()` pin/forced-refresh semantics (┬º8.3/┬º8.4).
- Replace the coarse `tar_combine()` step barriers with **per-child Option-3 upstream edges** (┬º6): each child shard target names only the specific parent shard target(s) its tasks read, computed at sourcing time, so a one-parent rewrite invalidates exactly the child shards that read it.
- Settle the deferred ┬º8 `data`-step-vs-fold decision as a consequence of the pinned model, and record the dual hazard the model must guard (a widened `n_max` must change the `sample` shard it feeds).

**Non-Goals:**

- The Hive write/read, `format = "file"` outputs, the m:n `read_parent_shards()` loop, `scenario_results_dir()` layout keying, byte-equivalence to the baseline ÔÇö all **already landed**; not re-specified.
- The end-to-end *assertions* of incremental rebuild on path-axis growth (`path-axis-growth`), inner-axis growth (`shard-atomic-rewrite`), or minimal scenario-slice rebuild (`step-scenario-slice`). This change pins the **model** those tests assert *against*; the tests themselves stay in their own changes.
- The `tar_cue(depend = FALSE)` *recipe step* (`mixed-code-lockin`) and the completeness-assert-driven refresh loop (`shard-completeness-assert`). This change pins the cue **semantics** the model relies on; the standalone recipes/tests are those changes.

## Decisions

### Decision: invalidation model is content-hash over `format = "file"`, read as cache-by-existence

The storage half already chose `format = "file"` (each shard target returns its Parquet path; `targets` content-hashes the file). The ┬º8 "cache-by-existence vs. content-hash" fork is therefore resolved **in favour of content-hash, with cache-by-existence as its observable consequence**: a shard is up to date iff (a) its output Parquet exists and (b) the bytes its body depends on ÔÇö its task-row set, the minimal scenario slice it reads, and the **named parent shard targets** it reads ÔÇö are unchanged. A shard whose Parquet is missing rebuilds (existence); a shard whose inputs changed rebuilds (content-hash); a recomputed shard whose bytes are byte-identical to before leaves its *dependents* skipped (value-propagation, confirmed by the targets-lab invalidation spike, ┬º8.2). The two framings are the same mechanism viewed from the output side (does the file exist / has it changed) vs. the input side (have the dependencies changed). *Alternative considered:* a directory-hash / manifest-driven cache-by-existence that ignores `targets`' content graph ÔÇö rejected; it duplicates what `format = "file"` + the content graph already give and loses `targets`' precise dependent tracking (┬º6 "no directory-hash indirection needed").

### Decision: per-child Option-3 upstream edges replace the coarse `tar_combine()` barrier

The shipped factory orders `fit` after `sample_done` and `hc` after `fit_done` ÔÇö a single barrier target combining *all* upstream shards. That barrier is the coarse-invalidation culprit: a child referencing it depends (transitively) on every upstream shard, so any upstream rewrite invalidates the child. The fix is ┬º6's "precise upstream edges": for each child shard, name **only** the specific parent shard target(s) its tasks read. The parent set is exactly what `read_parent_shards()` already computes to *read* them ÔÇö `unique(path_key(tasks, scenario$partition_by[[parent]]))` ÔÇö so the edge is the *names* of those parent path cells' targets, derived at sourcing time from the same shard tables `tar_map()` consumes. Concretely, a sourcing-time helper maps each child shard row to the vector of parent `*_step` target names its tasks reference, and the child's `tar_target_raw()` command references those names as symbols (so `targets` records the edges) before calling `ssd_run_<step>_step(...)`. The `tar_combine()` barriers are dropped as the *invalidation* wiring (the read-by-partition-path contract is unchanged; the runner still opens the same parent Parquets). Result: rewriting one parent shard invalidates the set of child shards that read it ÔÇö the **m:n** fan-in the ┬º8/┬º6 model requires ÔÇö not the whole step.

*Why this is the m:n-correct edge.* The relationship is m:n: a coarser parent keeps an axis the child shards on (one parent ÔåÆ many children), or a coarser child spans many parents (one child ÔåÆ many parents). Naming the *distinct projected parent set* per child captures both directions exactly: each child names all parents it reads; each parent is named by every child that reads it. A single named edge (the degenerate 1:1 case ┬º6 sketches as "name its one upstream data shard") is the special case when the projection yields one parent.

*Alternative considered:* keep the `tar_combine()` barrier but rely on value-propagation to skip unchanged children ÔÇö rejected; the barrier still forces every child *out of date* (it must at least re-check), and a child reading an unchanged parent should not even be considered stale. *Alternative considered:* dynamic `pattern = map` edges ÔÇö rejected; the shard set is static (known at sourcing), and ┬º6/`task-shards` commit to static branching with named, addressable shard targets.

### Decision: `tar_cue()` pin/forced-refresh semantics are part of the pinned model

The model's ┬º8.3/┬º8.4 recipes need defined cue behaviour, so this change states it as a requirement rather than leaving it to chance. `tar_cue(depend = FALSE)` on a step target pins it against upstream **dependency/code** changes (an edited `_state` function, a bumped `ssdtools`), so trusted shards are not rebuilt by a code edit ÔÇö **with the carve-outs the targets-lab confirmed (┬º8.3):** the target still rebuilds if its own `format = "file"` output is missing, if its task-table grouping changes (so path-axis/inner-axis growth still work under the pin), or if it previously errored (a ┬º6.2 short/failed shard retries). `tar_invalidate(names = ...)` (or `unlink()` of the Parquet) forces a refresh of chosen shards, overriding the pin ÔÇö the ┬º8.4 fix-and-refresh move. This change pins the **semantics**; the standalone cue *recipe* and its tests are `mixed-code-lockin`.

### Decision: settle `data`-step-vs-fold ÔÇö keep the fold

Under the pinned content-hash model the deferred ┬º8 decision resolves to **keep the `fit`-inline `head(sample, nrow)` fold** (`task-list-loop-baseline-fold`); no explicit `data` shard is reinstated. A `fit` shard is keyed by `fit_id`, which includes `nrow`, so extending `nrow` mints new `fit` shards and leaves existing ones cached ÔÇö the fold is sufficient and a materialised `data` checkpoint would be redundant. The dual hazard ┬º8 flags ÔÇö growing `n_max` (extending `nrow` upward) must not leave a stale short `sample` draw ÔÇö is handled by the model, not a `data` step: a widened `n_max` changes the `sample` shard's task row (`n_max` is a `sample` task column), so the `sample` shard's bytes change, and the **per-child edge** propagates that change to exactly the `fit` shards that read the widened draw. *Why not reinstate `data`:* the ┬º8 buffering-checkpoint argument applies only under content-hash propagation *without* a precise key ÔÇö i.e. when `fit` depends on the whole `sample` *value* and a wider draw cascades into every `fit` branch. Per-child edges + the `nrow`-keyed `fit_id` give the precise propagation a `data` checkpoint was meant to recover, so the checkpoint is unnecessary. The four-step ┬º8 examples are reconciled to the three-step (`sample`/`fit`/`hc`) fold by this decision.

### Decision: `summary` fan-in is unchanged

`summary` reads the result directories from disk (a directory read carries no `targets` edge), so it does not depend on each shard target's value. It is ordered after the shards; with the barriers dropped as the invalidation mechanism, `summary` is ordered via a lightweight all-`hc`-shards reference (or a retained terminal combine used **only** for ordering `summary`, never for inter-shard invalidation). Its value-propagation behaviour (re-runs when a shard it reads changed bytes; skips otherwise) is the existing ┬º8.2 behaviour and is not changed here.

## Risks / Trade-offs

- **Sourcing-time cost of computing per-child edges** ÔåÆ the projection is the same `path_key()` call `read_parent_shards()` already does at run time, run once per child shard at sourcing; cheap for the static-branching scale `task-shards` targets (the bounded-exception large-fan-out case opts into dynamic branching anyway, ┬º6).
- **Edge correctness vs. read correctness must agree** ÔåÆ the named parent set MUST equal the set `read_parent_shards()` opens, or a child could read a parent it does not depend on (stale read) or depend on a parent it does not read (spurious rebuild). Mitigated by deriving both from the *same* `unique(path_key(tasks, partition_by[[parent]]))` projection ÔÇö one source of truth ÔÇö and asserting equality in a test.
- **Dropping the `tar_combine()` barrier could lose `summary` ordering** ÔåÆ keep a minimal ordering reference for `summary` only (it reads the directory, so it needs an order edge but not per-shard value edges); the inter-shard invalidation no longer routes through any barrier.
- **`error = "null"` interaction with the cue pin** ÔåÆ an errored shard reruns regardless of the pin (┬º8.3 carve-out), which is the desired ┬º6.2/┬º8.4 behaviour; called out so it is not mistaken for a pin failure.

## Open Questions

- **`summary` ordering mechanism.** Whether to order `summary` via a retained terminal `tar_combine()` (used only for ordering, not invalidation) or via a direct reference to the `hc` shard targets is an implementation choice; both give the same value-propagation behaviour. Resolved during implementation.
- **Per-child edge target-name rendering.** The exact `tar_map()` name suffix for a parent shard target (so a child can name it) follows `scenario_partition_axes(scenario, parent)$path`; confirm the name-rendering matches `tar_map(names = ...)` for all default layouts during implementation.
